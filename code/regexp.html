First Name

1. Basic Regex

^[A-Za-z]+$

Valid: John

Invalid: John123

How it works: Allows only alphabets from start (^) to end ($). At least one letter required.

2. Business Regex

^[A-Z][a-z]{1,29}$

Valid: John

Invalid: john, J

How it works: Must start with uppercase letter followed by 1–29 lowercase letters. Minimum 2 characters, maximum 30.

3. Production Regex

^[A-Z][a-z]{1,29}(?:[-'][A-Z][a-z]+)?$

Valid: Mary, O'Neil, Anne-Marie

Invalid: mary

How it works: Proper capitalization required. Optionally allows hyphen or apostrophe with correct format.

Simple Explanation

1) ( ) → Capturing Group

Basic Example

/(cat)/

Text: "I love cats"

Output: Match "cat"

Captured Group 1: "cat"

Explanation: The parentheses capture the word "cat".

Real-World Example

/(\d{4})-(\d{2})-(\d{2})/

Text: "2026-02-27"

Output: Group 1: 2026 Group 2: 02 Group 3: 27

Explanation: Used to extract year, month, day.

Invalid Example

/(cat)/

Text: "dog"

Output: No match

Good Pattern

/(\w+)@(\w+)\.(\w+)/

Captures email parts.

Bad Pattern

/(.+)/

Too broad, captures everything.

2) (?: ) → Non-Capturing Group

Basic Example

/(?:cat|dog)/

Text: "cat"

Output: Matches "cat"

Explanation: Group used for logic, not captured.

Difference Example

/(cat|dog)/

Captures value.

/(?:cat|dog)/

Does NOT capture value.

Good Pattern

/https?:\/\/(?:www\.)?\w+\.\w+/

Bad Pattern

/(www\.)?/

Unnecessary capture.

3) (?<name> ) → Named Capture (.NET, JS)

Basic Example

/(?<year>\d{4})/

Text: "2026"

Output: year = 2026

Real-World Example

/(?<first>\w+)\s(?<last>\w+)/

Text: "John Smith"

Output: first = John last = Smith

Invalid Example

/(?<123>\w+)/

Invalid name. Names must start with letters.

4) (?P<name> ) → Named Capture (Python)

Example

/(?P<word>\w+)/

Text: "Hello"

Output: word = Hello

Invalid Example

/(?P<>\w+)/

No name provided → Error.

5) \1 → Backreference

Basic Example

/(cat)\1/

Text: "catcat"

Output: Match

Explanation: \1 refers to first captured group.

Invalid Example

/(cat)\1/

Text: "catdog"

Output: No match

Real-World Example

/(\w+)\s\1/

Text: "bye bye"

Output: Matches repeated word.

6) \k<name> → Named Backreference

Example

/(?<word>\w+)\s\k<word>/

Text: "hello hello"

Output: Match

Invalid Example

/(?<word>\w+)\s\k<other>/

No group named "other" → Error.

7) (?| ) → Branch Reset Group (PCRE)

Definition

Branch reset allows group numbers to restart in each branch.

Example

/(?|(cat)|(dog))/

Text: "dog"

Output: Group 1 = dog

Explanation: Both branches reset numbering to group 1.

Without Branch Reset

/(cat)|(dog)/

Group 1 = empty Group 2 = dog

Invalid Example

/(?|cat|dog)/

Must contain capturing groups inside.

Final Output Explanation

Capturing groups store matched text. Non-capturing groups group logic without storing. Named groups improve readability. Backreferences allow repeated matching. Branch reset is advanced and used in PCRE engines. These are very important in interviews, especially for extracting values and validating repeated patterns.

Lookarounds (Zero-Width Assertions)

Definition

Lookarounds are special patterns that check something before or after a word without including it in the final match. They are called "zero-width" because they do not consume characters.

Why We Use It

We use lookarounds when we want to: - Match something only if a condition is true - Exclude certain patterns - Validate format without capturing extra text

Syntax

(?= )  (?! )  (?<= )  (?<! )

Simple Explanation

Lookahead checks to the right. Lookbehind checks to the left. They test a condition but do not include it in the match.

1) (?= ) → Positive Lookahead

Basic Example

/\d(?=px)/

Text: "5px"

Output: Matches "5"

Explanation: Matches digit only if followed by "px".

Invalid Example

/\d(?=px)/

Text: "5em"

Output: No match

Explanation: Not followed by "px".

Real-World Example

/\w+(?=@gmail\.com)/

Text: "user@gmail.com"

Output: Matches "user"

Explanation: Extract username only if email is Gmail.

Good Pattern

/\d+(?=\sUSD)/

Extracts amount before currency.

Bad Pattern

/\d+USD/

This includes "USD" in match unnecessarily.

2) (?! ) → Negative Lookahead

Basic Example

/\d(?!px)/

Text: "5em"

Output: Matches "5"

Explanation: Matches digit only if NOT followed by "px".

Invalid Example

/\d(?!px)/

Text: "5px"

Output: No match

Real-World Example

/^(?!.*admin).*/

Text: "user123"

Output: Match

Text: "adminUser"

Output: No match

Explanation: Blocks usernames containing "admin".

Good Pattern

/\b\w+\b(?!\.)/

Bad Pattern

/\b\w+\b[^.]/

Incorrect logic.

3) (?<= ) → Positive Lookbehind

Basic Example

/(?<=\$)\d+/

Text: "$100"

Output: Matches "100"

Explanation: Matches digits only if preceded by "$".

Invalid Example

/(?<=\$)\d+/

Text: "100"

Output: No match

Real-World Example

/(?<=ID:)\d+/

Text: "ID:3456"

Output: Matches "3456"

Good Pattern

/(?<=https:\/\/)\w+\.\w+/

Bad Pattern

/https:\/\/(\w+\.\w+)/

Captures extra text unnecessarily.

4) (?<! ) → Negative Lookbehind

Basic Example

/(?<!\$)\d+/

Text: "100"

Output: Matches "100"

Text: "$100"

Output: No match

Explanation: Matches digits NOT preceded by "$".

Real-World Example

/(?<!Mr\.)\s\w+/

Text: "Mr. Smith"

Output: No match for space before Smith

Explanation: Avoids matching names after "Mr."

Valid Example

/(?<!#)\b\w+/

Matches words not starting with #.

Invalid Example

/(?<!\d+)\w+/

Some engines do not allow variable-length lookbehind.

Final Output Explanation

Positive lookahead checks what comes next. Negative lookahead blocks certain next patterns. Positive lookbehind checks what comes before. Negative lookbehind blocks certain previous patterns. They do not consume characters. They are very important in interviews for validation, filtering, and advanced pattern matching.

Inline Modifiers (Flags inside regex)

Definition

Inline modifiers are special flags written inside the regex pattern. They change how the pattern behaves.

Why We Use It

We use them when we want to control behavior like case sensitivity or multiline matching without changing global settings.

Syntax

(?i) (?m) (?s) (?x) (?U) (?-i) (?imxs)

Simple Explanation

They turn special modes ON or OFF inside the pattern itself.

1) (?i) → Case Insensitive

Basic Example

/(?i)cat/

Text: "CAT"

Output: Match

Explanation: Ignores uppercase/lowercase.

Invalid Example

/cat/

Text: "CAT"

Output: No match

Real-World Example

/(?i)error/

Matches: "Error", "ERROR", "error"

2) (?m) → Multiline

Basic Example

/(?m)^cat/

Text:

dog
cat

Output: Matches "cat"

Explanation: ^ works on each line.

Without Multiline

/^cat/

Output: No match

3) (?s) → Dot Matches Newline

Example

/(?s)a.*b/

Text:

a
b

Output: Match

Explanation: Dot includes newline.

Without (?s)

/a.*b/

Output: No match

4) (?x) → Ignore Whitespace

Example

/(?x)
\d+   # digits
\s
\w+   # word
/

Matches number and word with space.

Explanation: Spaces and comments ignored.

5) (?U) → Ungreedy (PCRE)

Example

/(?U)<.*>/

Text: "<b>bold</b>"

Output: Matches "<b>"

Explanation: Makes quantifiers lazy by default.

6) (?-i) → Turn Off Flag

Example

/(?i)cat(?-i)dog/

Text: "CATdog"

Output: Match

Text: "CATDOG"

Output: No match

Explanation: dog part is case-sensitive.

7) (?imxs) → Multiple Flags

/(?im)^cat.*dog$/

Text:

CAT
something dog

Output: Matches across lines.

Final Output Explanation

Inline modifiers control behavior inside the pattern. They are powerful for fine-tuning matching rules.

Regex Flags (Outside Pattern)

Definition

Flags placed outside the pattern change how the whole regex behaves.

Why We Use It

They help control global matching, case sensitivity, multiline behavior, and more.

Syntax

/pattern/flags

Simple Explanation

Flags come after the closing slash and apply to entire regex.

1) g → Global

/cat/g

Text: "cat cat"

Output: Matches both "cat"

Explanation: Finds all matches.

2) i → Case Insensitive

/cat/i

Text: "CAT"

Output: Match

3) m → Multiline

/^cat/m

Matches "cat" at start of any line.

4) s → Dotall

/a.*b/s

Dot includes newline.

5) u → Unicode

/\u{1F600}/u

Matches 😀 emoji correctly.

6) y → Sticky (JavaScript)

/cat/y

Matches only at current position.

Useful in token parsing.

7) x → Extended

/\d+ \w+/x

Whitespace ignored in some engines.

8) d → Indices (JavaScript ES2022)

/cat/d

Returns match index positions.

Final Output Explanation

Global flag finds all matches. i ignores case. m enables multiline anchors. s allows dot to match newline. u enables Unicode mode. y forces sticky matching. x ignores whitespace. d returns match indices. Understanding flags is very important for interview questions and real-world text processing.

Unicode & Property Classes

Definition

Unicode property classes match characters based on their type like letter, number, symbol, or script. They work only in Unicode-aware engines (like JavaScript with u flag, PCRE, .NET).

Why We Use It

We use them when working with international text, emojis, different languages, or advanced validation.

Syntax

\p{L} \p{N} \p{P} \p{S} \p{Z} \P{L} \p{Script=Latin} \p{Emoji}

Simple Explanation

\p means “match this Unicode category”. \P means “NOT this category”.

1) \p{L} → Any Letter

/\p{L}+/u

Text: "Hello नमस्ते 123"

Output: Matches "Hello" and "नमस्ते"

Explanation: Matches letters from any language.

Invalid Example

/\p{L}+/u

Text: "123"

Output: No match

2) \p{N} → Any Number

/\p{N}+/u

Text: "Year 2026"

Output: Matches "2026"

3) \p{P} → Punctuation

/\p{P}+/u

Text: "Hello, world!"

Output: Matches "," and "!"

4) \p{S} → Symbol

/\p{S}+/u

Text: "$100 + €50"

Output: Matches "$", "+", "€"

5) \p{Z} → Separator

/\p{Z}+/u

Text: "Hello World"

Output: Matches space

6) \P{L} → Not Letter

/\P{L}+/u

Text: "ABC123!"

Output: Matches "123!"

7) \p{Script=Latin}

/\p{Script=Latin}+/u

8) \p{Emoji}

Final Output Explanation

Unicode property classes help handle global languages, emojis, and advanced text validation. Very important for international applications.

Conditional & Advanced Constructs

Definition

Advanced constructs control regex behavior like condition checking, recursion, and backtracking control. Mostly available in PCRE, .NET, or advanced engines.

Why We Use It

Used in complex validation, parsing nested patterns, performance optimization, and interview-level problems.

Syntax

(?(condition)yes|no)
(?> )
(*FAIL)
(*SKIP)
(*PRUNE)
(*THEN)
(?R)
(?1)

Simple Explanation

These patterns give regex programming-like power.

1) (?(condition)yes|no) → Conditional

/(a)?b(?(1)c|d)/

Text: "abc"

Output: Match

Text: "bd"

Output: Match

Text: "ab d"

Output: No match

Explanation: If group 1 exists, match "c", otherwise match "d".

2) (?> ) → Atomic Group

/(?>a+)a/

Text: "aaaa"

Output: No match

Explanation: Atomic group prevents backtracking.

3) (*FAIL) → Force Fail

/cat(*FAIL)|dog/

Text: "cat"

Output: No match

Text: "dog"

Output: Match

4) (*SKIP) → Skip Match

/ignore(*SKIP)(*FAIL)|\w+/

Text: "ignore this word"

Output: Matches "this" and "word"

5) (*PRUNE) → Backtracking Control

/a(*PRUNE)b|ac/

Text: "ac"

Output: Match "ac"

6) (*THEN) → Control Flow

/a(*THEN)b|ac/

Text: "ac"

Output: Match "ac"

7) (?R) → Recursive Pattern

/\((?:[^()]++|(?R))*\)/

Text: "(a(b)c)"

Output: Full match

Explanation: Recursively matches nested parentheses.

8) (?1) → Subroutine Call

/(a(b))(?1)/

Text: "ab ab"

Output: Matches repeated subpattern

Final Output Explanation

Conditional patterns allow logic decisions. Atomic groups improve performance. FAIL, SKIP, PRUNE, THEN control backtracking. Recursive and subroutine calls allow nested pattern matching. These are advanced interview-level regex features.

Character Class Special Cases

Definition

These are special symbols used inside square brackets [ ]. They change how character classes behave.

Why We Use It

We use them to match ranges, exclude characters, or combine character sets in advanced ways.

Syntax

[^ ]   [a-z]   [a-z&&[def]]   [a-z~~[aeiou]]

Simple Explanation

Inside brackets: ^ at first position means NOT. - defines range. && means intersection (common part). ~~ means symmetric difference (exclusive part).

1) ^ → Negation (only first position)

Basic Example

/[^a]/

Text: "apple"

Output: Matches "p", "p", "l", "e"

Explanation: Matches anything except "a".

Valid Example

/[^0-9]/

Text: "abc123"

Output: Matches "a", "b", "c"

Invalid Example

/[a^]/

Text: "a^"

Output: Matches "a" and "^"

Explanation: ^ is not first position, so treated as normal character.

Good Pattern

/^[^0-9]+$/

Bad Pattern

/[^]/

Matches almost everything, too broad.

2) - → Range

Basic Example

/[a-z]/

Text: "Hello"

Output: Matches "e", "l", "l", "o"

Explanation: Matches lowercase letters.

Valid Example

/[0-9]/

Text: "A1B2"

Output: Matches "1", "2"

Invalid Example

/[z-a]/

Output: Error (invalid range)

Good Pattern

/^[A-Z][a-z]+$/

Bad Pattern

/[A-z]/

Explanation: Also includes unwanted ASCII characters.

3) && → Intersection (Java)

Basic Example

/[a-z&&[def]]/

Text: "abcdef"

Output: Matches "d", "e", "f"

Explanation: Matches letters that are both in a-z and in def.

Invalid Example

/[a-z&&[0-9]]/

Text: "abc123"

Output: No match

Explanation: No overlap between letters and digits.

Good Pattern

/[A-Za-z&&[^aeiouAEIOU]]/

Matches consonants only.

Bad Pattern

/[a-z&&]/

Invalid syntax.

4) ~~ → Symmetric Difference (Some Engines)

Basic Example

/[a-z~~[aeiou]]/

Text: "abcdef"

Output: Matches consonants only (b, c, d, f)

Explanation: Removes vowels from a-z.

Invalid Example

/[a-z~~]/

Output: Error

Good Pattern

/[0-9~~[2468]]/

Matches odd numbers.

Bad Pattern

/[~~]/

Invalid usage.

Final Output Explanation

^ excludes characters. - creates ranges. && finds common characters. ~~ removes overlapping characters. These are important for precise character matching.
