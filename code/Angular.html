1. Angular Introduction

1.1 What is Angular?

Angular is an open-source front-end web application framework developed and maintained by Google. It is mainly used to build modern, interactive, and dynamic Single Page Applications (SPA).

In simple words, Angular helps developers create websites and web applications where the page can change and update without reloading the entire browser window. Instead of creating many separate HTML pages, Angular creates one application that dynamically updates different parts of the screen.

Simple Definition

Angular is a TypeScript-based framework used to build fast, dynamic, component-based web applications.

Real Life Example

Imagine you are using an online banking website. You click:

  • Dashboard
  • Account Details
  • Transaction History
  • Fund Transfer

In older websites, every click would reload the entire page from the server. The browser would discard the old page and download a completely new HTML page.

In Angular, only the required part of the screen changes. The menu, header, footer, and other common elements remain the same, making the application much faster and smoother.

Why was Angular created?

As web applications became larger, managing hundreds of HTML files and JavaScript functions became difficult. Developers needed a structured way to organize code, reuse UI elements, communicate with APIs, and maintain very large applications.

Angular solves these problems by introducing concepts such as:

  • Components (small reusable UI blocks).
  • Services (shared business logic and API calls).
  • Routing (navigation without page refresh).
  • Dependency Injection (automatic object creation and sharing).
  • Forms and Validation.
  • HTTP communication with backend APIs.
  • Reactive programming using RxJS and Observables.

What can be built using Angular?

Angular is used to build many different types of web applications:

  • Banking and Financial Applications.
  • Hospital Management Systems.
  • E-Commerce Websites.
  • ERP (Enterprise Resource Planning) Systems.
  • CRM (Customer Relationship Management) Applications.
  • Admin Dashboards.
  • Inventory and Warehouse Management Systems.
  • Educational Portals and Learning Platforms.

  • Display and update data on the screen.
  • Validate user input.
  • Implement routing and navigation.
  • Handle login, logout, and authentication flows.
  • Coordinate with backend developers.

1.2 Why Angular?

Before learning Angular, an important question is: "Why was Angular created? Why can't we simply build websites using HTML, CSS, and JavaScript?"

The answer is that HTML, CSS, and JavaScript are enough for small websites, but as applications become larger, managing everything manually becomes difficult. Angular provides a structured way to build, organize, and maintain large web applications.

What is the Problem with Plain HTML, CSS, and JavaScript?

Imagine you want to build a simple online student management system. The application contains:

  • Login Page
  • Dashboard
  • Student List
  • Add Student Form
  • Edit Student Page
  • Profile Page
  • Settings Page

  • Which file calls the backend API.

In plain JavaScript, you may use global variables or repeatedly fetch the same information.

Does Angular replace the backend?

Can Angular directly connect to MySQL or PostgreSQL?

1.3 Single Page Application (SPA)

One of the most important concepts in Angular is the Single Page Application (SPA). Before learning components, routing, or services, it is necessary to understand what an SPA is and why Angular is designed around this idea.

What is a Single Page Application?

A Single Page Application (SPA) is a web application that loads a single HTML page when the application starts and then dynamically updates only the required parts of the screen without reloading the entire page.

Instead of downloading a new HTML page every time the user clicks a link, Angular replaces only the necessary component inside the current page.

Definition: A Single Page Application is a web application where one main page is loaded initially, and subsequent navigation updates only parts of the user interface without performing a full browser page reload.

What Happens in a Traditional Website?

In older websites, every menu click requests a completely new page from the server. 

User opens: www.example.com
        ↓
Server sends home.html
        ↓
Browser displays page

User clicks "Student List"
        ↓
Browser requests student.html
        ↓
Server sends student.html
        ↓
Entire page reloads

User clicks "Profile"
        ↓
Browser requests profile.html
        ↓
Server sends profile.html
        ↓
Entire page reloads again

Every navigation causes:

  • A new HTTP request for the page.
  • A complete browser refresh.
  • Rebuilding of the entire UI.
  • Loss of the previous page state.

What Happens in an Angular SPA?

In Angular, only one main page is loaded initially. After that, Angular itself controls navigation inside the browser. 

User opens: www.example.com
        ↓
Server sends Angular application
(index.html + JavaScript bundles)
        ↓
Angular starts

User clicks "Student List"
        ↓
Angular Router loads StudentComponent
        ↓
Only page content changes
        ↓
No full page reload

User clicks "Profile"
        ↓
Angular Router loads ProfileComponent
        ↓
Only page content changes
        ↓
Header, menu, and footer remain the same

SPA vs Traditional Multi Page Application (MPA)

User Experience Slower. Smoother and faster.
Navigation Browser requests new HTML. Angular Router changes component.
Header/Footer Reload every time. Remain on screen.

  • Click Dashboard → Dashboard Component appears.
  • Click Accounts → Accounts Component appears.
  • Click Transfer → Transfer Component appears.

Angular simply replaces the center portion of the page instead of reloading everything.

How Does Angular Do This?

Angular uses a feature called Routing. The Router decides which component should be displayed based on the URL.

You will later learn about:

  • routerLink
  • <router-outlet>
  • Route Configuration
  • Route Guards

But for now, just remember that Angular watches the URL and loads the appropriate component into a placeholder called router-outlet.

What is Actually Loaded from the Server?

Many beginners think Angular downloads a new HTML page every time. Actually, the server sends the Angular application only once.

        ▼
+------------------------------+

| - index.html                 |
| - Angular JavaScript files   |

+------------------------------+

        │
        ▼

Only API calls (such as getting student data or transaction history) go to the backend server.

SPA Does NOT Mean "No Server"

A common misunderstanding is:

"Since Angular does not reload pages, does it stop talking to the server?"

The answer is No.

Angular still communicates with the backend whenever it needs data. The difference is that it requests JSON data instead of requesting complete HTML pages. 

Traditional Website

Browser ─────► Server
        Request: student.html
Browser ◄───── Server
        Response: Complete HTML Page


Angular SPA

Browser ─────► Backend API
        Request: /api/students
Browser ◄───── Backend API
        Response: JSON Data

Angular updates UI automatically.

Advantages of SPA

  • Fast navigation between pages.
  • No full browser refresh.
  • Better user experience.
  • Reusable UI components.
  • Reduced server load because HTML pages are not repeatedly generated.
  • Feels similar to a desktop application.

Disadvantages of SPA

  • Initial loading may take slightly longer because the Angular application must be downloaded.
  • JavaScript must be enabled in the browser.
  • The application architecture is more complex than a simple static website.
  • Developers need to understand routing, components, and state management.

SPA Analogy

Imagine a hotel.

  • Traditional Website: Every time you want to visit another room, the entire hotel building is demolished and rebuilt.
  • Angular SPA: The hotel building stays the same, and you simply walk from one room to another.

The building (browser page) remains. Only the room (component) changes.

Relationship Between SPA and Components

Single Browser Page

Angular applications are built from many small components. SPA works because Angular can replace one component while leaving the others untouched.

Common Beginner Doubts

Is Angular only for SPA?

Angular is mainly designed for building Single Page Applications, although it can also be used in hybrid approaches.

If the page never reloads, how does new data appear?

Angular calls backend APIs using HttpClient. The server returns JSON data, and Angular updates the displayed information automatically.

If I type a new URL in the browser, does Angular still work?

Yes. Angular Router reads the URL and loads the appropriate component without downloading a completely new page.

Does SPA mean only one HTML file exists?

Not exactly. During execution, Angular mainly works from one root HTML page (index.html), but your project still contains many component templates and files.

Quick Revision

  • SPA stands for Single Page Application.
  • An SPA loads the application once and updates only required content.
  • Angular uses Routing to switch between components.
  • No full browser page refresh happens during navigation.
  • Angular requests JSON data from backend APIs instead of complete HTML pages.
  • Header, menu, and footer can remain while only the main content changes.
  • SPA provides a smoother and faster user experience.

1.4 Angular vs Traditional Web Applications

To understand Angular deeply, it is important to compare it with the traditional way of building websites. Many beginners think Angular is simply "HTML + CSS + JavaScript", but Angular follows a completely different architecture.

What is a Traditional Web Application?

A traditional web application is a website where the server generates and sends a complete HTML page every time the user requests a new page. 

User clicks "Student List"
        ↓
Browser sends request
        ↓
Server processes request
        ↓
Server creates complete HTML page
        ↓
Browser receives HTML
        ↓
Entire page reloads

In this model, the server is responsible for both:

  • Business logic.
  • Generating the HTML shown to the user.

What is an Angular Application?

In an Angular application, the frontend and backend have separate responsibilities.

  • Angular handles the user interface (UI).
  • The backend handles business logic and database operations.
  • The backend returns JSON data instead of complete HTML pages.
User clicks "Student List"
        ↓
Angular Router loads StudentComponent
        ↓
StudentComponent calls StudentService
        ↓
StudentService sends HTTP request
        ↓
Backend API processes request
        ↓
Database returns data
        ↓
Backend sends JSON response
        ↓
Angular updates UI automatically

2.1 What is TypeScript?

TypeScript is a programming language developed by Microsoft. It is a superset of JavaScript, which means:

Every valid JavaScript code is valid TypeScript code, but TypeScript adds extra features on top of JavaScript.

Angular is built using TypeScript, so understanding it is very important before going deeper into Angular concepts.

Why TypeScript?

JavaScript is flexible, but that flexibility becomes a problem in large applications. TypeScript solves these problems by adding structure and safety.

Problem with JavaScript

JavaScript allows changing data types freely, which can lead to runtime errors.

How TypeScript Fixes This


let value: number = 10;
value = "Hello";   // ERROR in TypeScript

TypeScript catches errors before running the code.

TypeScript Features

  • Static typing
  • Interfaces
  • Classes and OOP support
  • Better code organization
  • Compile-time error checking
  • Modern JavaScript features support

6. Angular Lifecycle Hooks

Angular Lifecycle Hooks are special methods that Angular calls automatically at different stages of a component’s life (from creation to destruction).

Think of a component like a human life cycle: birth → growth → updates → end. Angular gives hooks at each stage.

6.1 Constructor vs ngOnInit()

This is one of the most important interview concepts in Angular. Both are used during component creation but serve different purposes.

Constructor

Constructor is a TypeScript feature, not Angular-specific. It is executed when the class is created.

Used for:

  • Dependency injection (services)
  • Basic initialization

constructor(private service: UserService) {
  console.log("Constructor called");
}

ngOnInit()

ngOnInit is an Angular lifecycle hook. It runs after Angular has fully initialized the component.

Used for:

  • API calls
  • Loading data
  • Initializing component data

ngOnInit() {
  console.log("ngOnInit called");
  this.loadUsers();
}

Key Difference

  • Constructor → runs first (class creation time)
  • ngOnInit → runs after Angular initializes component
  • Constructor → no input bindings ready
  • ngOnInit → input data is available

Example Flow


Constructor → Dependency setup
     ↓
ngOnInit → Load data from API

6.2 ngOnInit()

ngOnInit is one of the most commonly used lifecycle hooks in Angular. It is called only once after component initialization.

Why ngOnInit is important?

Because Angular ensures all inputs and dependencies are ready before calling it.

Example: API Call


import { Component, OnInit } from '@angular/core';
import { HttpClient } from '@angular/common/http';

@Component({
  selector: 'app-users',
  templateUrl: './users.component.html'
})
export class UsersComponent implements OnInit {

  users: any[] = [];

  constructor(private http: HttpClient) {}

  ngOnInit() {
    this.http.get('https://jsonplaceholder.typicode.com/users')
      .subscribe(data => {
        this.users = data;
      });
  }
}

Template Example


<ul>
  <li *ngFor="let user of users">
    {{ user.name }}
  </li>
</ul>

Simple Meaning

ngOnInit is where you "start working with real data".

Angular provides multiple lifecycle hooks for different stages of a component.

1. ngOnChanges()

Called when input properties change from parent component.

2. ngDoCheck()

Called during every change detection cycle.

3. ngAfterViewInit()

Called after component’s view (HTML) is fully initialized.

4. ngAfterContentInit()

Called after external content (ng-content) is projected into component.

5. ngOnDestroy()

Called just before component is removed from DOM. 


ngOnDestroy() {
  console.log("Component destroyed");

  // Cleanup tasks
  // unsubscribe observables, remove timers
}

Use case: Prevent memory leaks

↓
ngAfterContentInit

NgRx (Redux Pattern for Angular)

NgRx is a state management library for Angular based on the Redux pattern.

Simple meaning: NgRx = “central brain that stores and manages all app data in one place”

Why NgRx is needed?

  • In large apps, data becomes hard to manage
  • Multiple components need same data
  • API data must be shared globally
  • To avoid duplicate API calls

NgRx Main Architecture


Component → Dispatch Action → Reducer → Store → Selector → Component
                    ↓
                 Effects (API calls)

1. Store

Store is a single global state container for the entire Angular application.

Simple meaning: Store = “big box where all app data is saved”

Example State


{
  user: { name: "Aathi", role: "admin" },
  cart: [],
  products: []
}

2. Actions

Actions are events that tell NgRx “something happened”.

Simple meaning: Action = “instruction or event trigger”

Example


import { createAction } from '@ngrx/store';

export const loadUsers = createAction('[User] Load Users');
export const loadUsersSuccess = createAction(
  '[User] Load Users Success'
);

Types of Actions

  • Load data
  • Success
  • Failure
  • Add / Update / Delete

3. Reducers

Reducers are pure functions that take current state + action and return new state.

Simple meaning: Reducer = “decision maker that updates state”

Example


import { createReducer, on } from '@ngrx/store';

export const initialState = {
  users: []
};

export const userReducer = createReducer(
  initialState,

  on(loadUsersSuccess, (state, action) => {
    return {
      ...state,
      users: action.users
    };
  })
);

Important Rule

  • Reducers must be PURE
  • No API calls inside reducer
  • Only state transformation

4. Effects

Effects handle side effects like API calls, logging, or async operations.

Simple meaning: Effects = “middle layer for API calls”

Example


import { Injectable } from '@angular/core';
import { Actions, createEffect, ofType } from '@ngrx/effects';
import { HttpClient } from '@angular/common/http';
import { map, switchMap } from 'rxjs/operators';

@Injectable()
export class UserEffects {

  constructor(private actions$: Actions, private http: HttpClient) {}

  loadUsers$ = createEffect(() =>
    this.actions$.pipe(
      ofType(loadUsers),
      switchMap(() =>
        this.http.get('api/users').pipe(
          map(users => loadUsersSuccess({ users }))
        )
      )
    )
  );

}

Flow


Action → Effect → API Call → Success Action → Reducer → Store

  selectUsers,

users$ = this.store.select(selectUserList);

4. Browser Cache (Deep JavaScript + Web Performance)

Browser Cache is a mechanism used by the browser to store files locally so that websites load faster on next visit.

Simple meaning: Cache = “browser memory that saves website files to avoid downloading again”

1. How Browser Cache Works Internally


First Visit:
Browser → Downloads HTML, CSS, JS, Images from server

Next Visit:
Browser → Uses saved cached files (no re-download)

This reduces: - Network usage - Load time - Server load

2. What gets cached?

  • HTML files
  • CSS files
  • JavaScript files
  • Images
  • Fonts
  • API responses (sometimes)

3. Types of Cache

1. Memory Cache

  • Stored in RAM
  • Very fast
  • Cleared when tab closes

2. Disk Cache

  • Stored in hard disk
  • Persists even after restart

4. Cache-Control (VERY IMPORTANT)

Server controls caching using HTTP headers. 


Cache-Control: max-age=3600

Meaning: Store file for 1 hour (3600 seconds)

5. Real Example (Network Request Flow)


First request:
GET /app.js → Download from server

Second request:
GET /app.js → Loaded from cache (FAST)

6. Cache Busting (IMPORTANT in Angular)

Sometimes cache causes old code to run.

Solution = change file version or hash. 


app.js?v=1
app.js?v=2

Angular automatically handles this using build hashes: 


main.8f3a1c.js
runtime.a91b2d.js

7. API Cache (Advanced Concept)

Sometimes API responses are also cached. 


GET /products
→ cached response returned instead of backend call

8. Angular Example (HTTP Cache Behavior)


this.http.get("api/products").subscribe(data => {
  console.log(data);
});

If cache is enabled, browser may not hit server again.

9. Force Refresh (Bypass Cache)

  • Ctrl + F5 → hard refresh
  • Shift + Reload → bypass cache

10. Cache Problems (VERY IMPORTANT)

  • Old JS file may run
  • UI not updating after deployment
  • API returns stale data

11. Cache Fix Strategies

  • File hashing (Angular default)
  • Versioning (v1, v2)
  • Disable cache for APIs
  • Service workers (PWA control)

12. Real Life Analogy

  • Cache = your room drawer
  • Frequently used items are stored there
  • So you don’t go to shop again and again

13. Key Points

  • Cache improves performance
  • Stores files locally in browser
  • Controlled by HTTP headers
  • Can cause outdated data issues
  • Angular handles cache using hashed builds
